**Self-Constructing Agents with Verified Skill Persistence: A Cold–Warm Evaluation Across Coding Domains**  
**Ihsan Alaeddin, Sazan Khalid**

**Abstract**

Language-model agents typically operate without persistent memory of how they solved past problems: every new task begins from a blank slate, and the engineering knowledge produced during one session evaporates when the session ends. When a user wants to re-prompt a specific skill, they end up explaining the context all over.

This paper presents a self-constructing agent architecture in which the agent writes, verifies, and accumulates a permanent library of executable Python skills as a by-product of normal task execution, and reuses them across future sessions. 

The architecture extends the open-ended action accumulation paradigm with four contributions: a four-gate verification pipeline that admits only skills which pass syntactic, executional, static-compliance, and adversarial-validation checks; a skill lifecycle layer that composes, evolves, and decays skills based on observed usage; an episodic memory layer that retrieves the most semantically similar past tasks before each new attempt; and a domain application targeting the November 2025 SWIFT ISO 20022 migration, an acute and well documented integration crisis in cross-border payments. 

We evaluate the system in a controlled cold–warm experiment against the FastAPI codebase, comparing an agent with no prior skills against an agent that has accumulated a library through twenty exploration tasks. The warm agent achieved a mean score of 2.67 out of 3.0 versus the cold agent's 2.00, a relative improvement of 33.3%, with gains observed across all three task tiers including structural retrieval (+1.00), pattern replication (+0.50), and novel synthesis (+0.50). 

Notably, only one of the six accumulated skills was directly invoked during the warm benchmark — suggesting that the architecture's benefit is multi-channel, deriving partly from accumulated context quality and episodic memory rather than purely from direct skill reuse. We discuss limitations including small sample size, single-model evaluation, and domain mismatch between exploration and benchmark, and we identify how each shaped the result.

1. **Introduction**

Most large-language-model agents currently in production are stateless. They receive a task, reason about it through a chain of tool calls or code execution steps, return an answer, and forget everything they learned in the process. The next time a similar task appears the entire reasoning trace has to be reconstructed. The institutional knowledge generated by an agent's effort accumulates only in the form of conversation logs, which no subsequent agent reads.

This is a bigger limitation than it appears. In any domain where the same kinds of problems recur, a stateless agent is structurally incapable of getting better at the work, because every session starts at the same baseline as the first. A consultant who has spent three years writing ISO 20022 message parsers becomes faster, more accurate, and more aware of edge cases over time; the agent does not. 

The asymmetry is not a matter of model size or prompt engineering. It is a structural consequence of the absence of cross-session persistent procedural memory. This paper studies a class of agents that do retain procedural memory in a specific form: as verified, executable Python functions saved to a local skill library. Each time the agent solves a task that produces reusable code, it writes that code as a function with metadata, runs a unit test against it, and persists it only if the test passes and a series of additional checks succeed. 

Future tasks can retrieve, invoke, compose, and evolve these skills. The library is the agent's accumulated procedural knowledge, encoded in a form that is verifiable rather than narrative. 

To concentrate this project into a field, we chose to integrate it into a financial-protocol field. SWIFT, the network connecting over eleven thousand financial institutions across more than two hundred countries, formally retired the legacy MT message format in November 2025 in favour of ISO 20022 MX. Every cross-border payment instruction must now be exchanged in the new structured-XML format. Industry surveys conducted ahead of the deadline projected that nearly five thousand member banks would not be fully compliant in time \[@swift\_migration\_survey\_2024\]. 

Eighty-three per cent of payment-modernisation projects are estimated to deliver late, eighty-seven per cent over budget, and eighty-three per cent failing original specifications  
\[@payment\_modernisation\_industry\_2025\]. The financial cost of the integration knowledge gap  
is not theoretical: each problematic project costs an average of approximately half a million US  
dollars, and the consulting industry that exists to bridge this gap has a structural disincentive to  
make its knowledge transferable.

A self-constructing agent that accumulates verified payment-protocol skills is, in principle, a partial solution to this problem. The skills it builds in response to one institution's integration tasks are applicable to the next institution's integration tasks. The knowledge does not walk out the door when a consultant rotates off an engagement. The library compounds across deployments rather than starting from zero each time.

The research questions of this paper are:

1. Does an agent that has accumulated a verified skill library perform measurably better on novel tasks than one starting cold?  
2. Where does the benefit come from — direct skill reuse, accumulated context quality, episodic memory, or some combination?  
3. Is the verification pipeline strict enough to keep the library trustworthy without being so strict that it suppresses skill formation entirely?

We answer the first two empirically, through a cold–warm comparison experiment described in Sections 5 and 8\. We answer the third through a discussion of skill formation rates, ComplianceShield outcomes, and adversarial-validation pass rates in Section 8\. Section 11 concludes by listing what would be required to extend this from a research artefact into a system that could be safely deployed against live payment infrastructure.

2. **Related Work**  
   

This architecture integrates three core research paradigms: reasoning scaffolds, code-driven actions, and persistent skill accumulation. To illustrate the evolution of these concepts, we focus on three seminal works:

1. ReAct: Reasoning and Acting in Language Models:  
   ReAct \[@yao\_react\_2022\] pioneered the interleaving of free-form reasoning traces with structured tool calls. By forcing a model to "think" before it "acts," ReAct established the standard scaffold for agentic planning. However, it treats the action space as a fixed library provided by the designer, limiting the agent to pre-defined tools rather than allowing it to evolve its own capabilities.  
     
2. CodeAct: Using Code as the Action Space:  
   CodeAct \[@wang\_executable\_2024\] argues that Python code is a more expressive and efficient action representation than JSON tool calls. By emitting and executing code directly, agents can handle complex logic—such as loops, nested functions, and variable reuse—within a single turn. This paradigm significantly reduces the total number of model calls required to solve multi-step tasks compared to traditional tool-calling frameworks.  
     
3. Voyager: Open-Ended Skill Acquisition  
   Voyager \[@wang\_voyager\_2023\] demonstrated that agents can autonomously author, verify, and store a procedural memory of skills. In a Minecraft environment, Voyager showed that an agent could accumulate a library of parameterized programs to solve increasingly difficult downstream tasks. Our work extends this "action accumulation" logic to financial protocols, adding rigorous four-stage verification gates and a decay-based lifecycle for skill archival.

3. **System Architecture**  
   

The system runs as a single Python process organised in seven layers. Each layer has a defined responsibility and a defined interface to its neighbours; each persistent store is held in a flat-file format readable by external tools.

**3.1 Layer 1 — Orchestration**

The orchestration layer ( run.py , 1629 lines) is the central task loop. It reads tasks from a queue file (one task per line), maintains all in-memory state for a session, and is responsible for assembling the system prompt before each LLM call and for routing the response to the correct downstream handler. The layer also owns the MCP client loop described in Section 3.5: when an LLM response contains a tool-call signal, the orchestration layer intercepts it, executes the tool against the registered server, injects the result back into the conversation, and re-calls the model.

**3.2 Layer 2 — LLM Reasoning**

The system uses a single LLM at the reasoning layer, configurable by environment variables ( Gemini ). For the experimental runs reported in Section 5 we used gemini-2.5-flashlite . The model receives the assembled system prompt and produces one of four block types in its response:

\---EXECUTE--- : a Python snippet to run inline in a sandboxed subprocess, with stdout returned to the conversation as further context. Execution is ephemeral; nothing is persisted.  
\---PY SKILL--- : a Python function with a unit test, intended to be saved as a reusable skill if it passes the verification pipeline.  
\---MCP TOOL--- : a Python function plus a JSON input/output schema, intended to be registered as an MCP tool.  
\---NEW TASK--- : a new task to append to the work queue, allowing the agent to generate follow-up work for itself.  
\---TOOL CALL--- , indicates a pause in reasoning rather than a final action: the agent has identified that it requires live external data before proceeding. The orchestration layer handles this signal as described in Section 3.5.

**3.3 Layer 3 — Verification Pipeline**

Every candidate skill passes through four sequential gates before being persisted to disk. Each gate has the authority to reject the skill, in which case the agent falls back to returning the raw LLM output as the task answer and no skill enters the library. 

**Gate 1** — AST validity. The skill source is parsed via Python's ast.parse . Syntax errors at this stage are common and cheap to detect; they are the most frequent cause of skill rejection.

**Gate 2** — Executional verification. The skill source and its unit test are written to a temporary file and executed in a subprocess with a thirty-second timeout. The skill is admitted only if the subprocess exits with return code zero and its stdout contains the literal string TEST PASSED .The decision to require an exact output marker rather than a non-zero exit code alone is deliberate: many models will write tests that use Python's assert statement without a print, succeed silently, and exit zero with no output. Requiring an explicit success print disambiguates "test passed" from "test never ran".

**Gate 3** — ComplianceShield (static analysis). The compliance.py module (143 lines) performs regex-based static analysis on the skill source. Two rule classes are applied. The first is a block list: hard-coded nine-digit numbers (potential ABA routing numbers), IBAN literals matching a country-prefixed pattern, hard-coded passwords or API keys assigned to string literals, and U.S. SSN-pattern strings cause the skill to be rejected outright with a BLOCK outcome. The second is a warn list: skills containing identifiers or function names from a curated audit-relevance vocabulary (e.g. account\_number , iban , creditor , parse\_pacs ) are checked for the presence of an audit hook ( logging.info , audit\_trail , compliance\_log ); skills missing such a hook persist with a WARN flag in their metadata. The shield runs on every candidate skill regardless of domain, but its rules are calibrated for the payment domain.

**Gate 4** — Adversarial validation. The validation\_agent.py module (121 lines) generates a second, independent test of the skill via an LLM call distinct from the one that authored the skill. The validating model receives only the skill's source and an instruction to inline the function into a fresh test harness containing six adversarial assertions targeting empty, None , boundary, verylarge, negative, unicode, and type-edge inputs. The harness must end with the literal string VALIDATION PASSED printed to stdout. If LLM generation is unavailable (rate limit, transient API error, four retries exhausted), the validation step returns a skipped status rather than a failure, on the principle that an opaque infrastructure failure should not block skill formation. This is a deliberate trade-off discussed in Section 9\.

**3.4 Layer 4 — Three Persistent Stores**

Three flat-file stores hold all cross-session states. The skill library ( skills\_py/ ) holds verified Python skills as individual files. Each file begins with a metadata header ( \# skill: , \# version: , \# tags: , \# verified: , \# protocol: , \# rail: , \# audit\_required: , \# last\_used: ) followed by the function source and its unit test. A manifest at skills\_py/index.json lists every active skill with its metadata, allowing the selection layer to scan the library without reading every file. Failing or evolved skills are moved to skills\_py/archive/ with a version suffix.

The episodic memory ( memory/episodes.jsonl ) is an append-only log of every completed task. Each line records the task text, outcome, skills used, whether composition or evolution occurred, the failure type if any, and a recovery outcome. Before each new task, the orchestration layer embeds the task text, computes cosine similarity against the embeddings of the most recent two hundred episodes, and injects the top three into the system prompt as context. When the embedding model is not available (network failure or library not installed), the system falls back to word-overlap scoring.

The tool registry ( tools/registry.json , managed by mcp\_client.py , 306 lines) records every MCP tool — both built-in and agent-built. Each entry holds the tool's name, input schema, output schema, source file (for agent-built tools), usage count, and observed failure rate.

**3.5 Layer 5 — Skill Lifecycle Mechanics**

Four mechanics keep the library accurate and lean over time. 

**Composition**: Before generating new code for a task, the orchestration layer retrieves the three highest-similarity skills from the library and asks the model whether it can solve the task by combining them. If yes, the composition is built, verified through the same four-gate pipeline, and saved with a composed\_from field listing its component skills. 

**Evolution**: When a skill's accumulated fail\_count reaches two and exceeds its success\_count , the model is asked to diagnose and patch the function. The patched version is verified through the full pipeline; on success, the previous version is archived as \<name\>\_v1.py and the new version replaces it with a bumped version number.

**Decay**: Each skill's relevance score is computed as success\_count / (1 \+ days\_since\_used × 0.1) . Below 0.5 with age over seven days, a skill is flagged as decaying and de-prioritised in selection. Below 0.1 with age over fourteen days, the skill is auto-archived. The active library remains lean rather than growing unboundedly.

**Failure** **recovery**: After every task attempt, a failure classifier grades the output on a three-point scale (0/1/2). If the grade is one or below, it categorises the failure as one of five types — hallucination , missing\_context , logical\_inconsistency , tool\_misuse , or stylistic\_misalignment — and applies a type-specific recovery strategy before marking the task failed. Hallucination triggers a constrained re-prompt restricted to confirmed file references. Missing context fetches additional repository files. Tool misuse re-reads the actual API signature from source. Logical inconsistency triggers a structured four-step self-verification prompt. Stylistic misalignment injects three real file examples as convention guidance. Each task gets at most one recovery attempt.

**3.6 Layer 6 — MCP Tool Calling**

The mcp\_client.py module implements a lightweight Model Context Protocol client. Four built-in tools are pre-registered: read\_file (returns up to four thousand characters of a local file), search\_code (keyword search across .py files), run\_python (executes a snippet in a subprocess), and list\_skills (returns the current skill manifest). Agent-built MCP tools are registered through the \---MCP TOOL--- block path described in Section 3.2.

When the LLM emits a \---TOOL CALL--- block, the orchestration layer parses the JSON payload, dispatches to the registered tool, captures the result, injects it back into the conversation as a TOOL RESULT \[\<tool\>\]: line, and re-calls the model. The model never executes tools directly; it signals intent and the orchestration layer fulfils the call. This separation is important for verifiability: a skill that depends on live data (e.g. an FX rate) accepts that data as a function parameter rather than fetching it internally, so the skill itself remains a pure function and is testable offline. The corresponding tool call sits at the orchestration layer where it can be mocked for testing or replaced with a different data source without touching the skill.

**3.7 Layer 7 — RAG Reference Index**

The rag.py module (287 lines) maintains a small corpus of seeded reference documents covering: ISO 20022 message-family overviews; the SWIFT MT-to-MX migration timeline; the November 2026 hybrid-address requirement; the NACHA ACH file format; ABA routing-number checksum algorithm; cross-rail reconciliation patterns; and standard ledger-entry field definitions. Documents are chunked, embedded with sentence-transformers/all-MiniLM-L6-v2 , and retrieved by similarity to the current task. The two highest-scoring chunks are injected into the system prompt alongside skills, episodes, and tool manifests. RAG is read-only from the agent's perspective; it represents passive reference knowledge rather than learned procedural skill.

The combination of all four context sources — episodic memory, skill library, tool manifests, and RAG — is reassembled fresh for every task. Nothing carries over in raw form between tasks; persistence lives entirely in the four flat-file stores.

4. **Models and Technologies**  
     
   The reported experiment uses Google's gemini-2.5-flash-lite as the reasoning model, accessed via the google-genai Python client. The choice was driven primarily by free-tier daily token allowance — the lite tier permits roughly 1.5 million tokens per day on Google AI Studio, against approximately 14,400 tokens per day for llama-3.3-70b-versatile on Groq's free tier — which determined whether a full cold–warm experiment could be completed in a single calendar day.  
   

Embedding for skill retrieval and episodic memory similarity uses sentencetransformers/all-MiniLM-L6-v2 , a 22-million-parameter sentence encoder running locally on CPU. Both subsystems fall back to keyword-overlap scoring when the embedding model is unavailable.

The agent has no dependency on a higher-order framework such as LangChain or LlamaIndex. The orchestration layer, the LLM client wrapper, the verification pipeline, the skill library, the lifecycle mechanics, the failure classifier, the MCP client, and the RAG layer are implemented directly and distributed across run.py, mcp\_client.py, rag.py, payment\_trace.py, compliance.py, and validation\_agent.py. 

The decision not to build atop a larger framework was deliberate; framework abstractions tend to obscure the verification and lifecycle mechanics that this work  
specifically aims to study.

A token-budget guard ( TokenBudget class, \~30 lines in run.py ) tracks estimated cost per call against a configurable USD limit and pauses execution before the limit is exceeded. The guard is necessary because the system's task loop can run unattended for hours, and an unbounded loop on a paid model is a meaningful financial risk.

External tooling is minimal: subprocess for skill verification and code execution; xml.etree.ElementTree for ISO 20022 parsing in the synthetic target repository; urllib.request (rather than a third-party HTTP client) for the FX-rate skill, to keep the skills\_py directory dependency-free and verifiable in any reasonable Python 3.11 environment.

The single optional third-party dependency in skills is sentence-transformers , isolated to the embedding subsystems.

5. **Data and Evaluation**  
     
   **5.1 The Cold–Warm Experimental Design**  
     
   The hypothesis under test is that an agent with an accumulated skill library performs

measurably better on novel tasks than one starting cold. The experimental design is a within subject comparison of two conditions on the same set of benchmark tasks.

In the cold condition, the skill library and episodic memory are wiped before the run. The agent then attempts six benchmark tasks in sequence with no prior procedural knowledge.

In the warm conditions, the skill library and episodic memory are wiped, then the agent first runs through twenty exploration tasks intended to populate the library. After exploration, the same six benchmark tasks are run, this time with whatever skills survive verification available for retrieval, composition, and reuse.

The benchmark and exploration sets are held strictly separate. The benchmark file is never modified between conditions. Both conditions use identical model, prompt scaffolding, retrieval parameters, and verification gates; the only difference is whether the skill library is empty or populated when the benchmark begins.

**5.2 The Benchmark Set**

The benchmark targets the open-source FastAPI codebase ( tiangolo/fastapi ), a reasonably mature Python project with non-trivial internal structure. Benchmark tasks are organised into three tiers of increasing difficulty, designed to surface different mechanisms by which a skill library might help.

Tier 1 — Structural retrieval (5 tasks; 2 used in this run). Read the codebase and report a fact: which modules import which Starlette names, which file defines FastAPI and which class it inherits from, what public methods does APIRoute define. A cold agent should be able to do  
these by reading files; the warm agent's advantage, if any, comes from prior context familiarity  
rather than skill substitution.

Tier 2 — Pattern replication (5 tasks; 2 used). Write new code that follows an existing convention precisely: a new APIRoute subclass that follows the constructor signature of the existing one, a Depends \-style helper modelled on param\_functions.py , a security-scheme class that mirrors APIKeyHeader but reads a query parameter. The warm agent should benefit from having seen analogous patterns during exploration.

Tier 3 — Novel synthesis (5 tasks; 2 used). Identify a pattern repeated across at least three files and abstract a reusable skill from it; spot a duplicated conditional in routing.py and refactor it; produce a minimal reproduction of FastAPI's request validation chain in fewer than sixty lines. These tasks ostensibly require composition or generalisation, not simple recall. In the reported run, six benchmark tasks were used (two per tier, sliced by per\_tier \= max\_bench // num\_tiers ), reflecting a budget-imposed reduction from the originally designed fifteen-task benchmark. The full benchmark file remains intact for replication.

**5.3 The Exploration Set**

The exploration set used in the reported run comprises twenty single-function ISO 20022 and ACH utility tasks ( exploration\_tasks\_simple.txt ): cents-to-dollars conversion, IBAN format  
normalisation, country-code extraction, ABA-checksum calculation, BIC-format validation,  
currency-symbol lookup, and so on. Each task specifies a function signature and an inline unit  
test that ends with print("TEST PASSED") , designed to be solvable by a small fast model and to  
exercise the verification pipeline reliably.

The exploration tasks were drawn from the payment domain rather than the FastAPI domain. This is a domain mismatch with the benchmark and is a meaningful methodological caveat that we examine in Section 8: it means the warm condition's accumulated skills are largely irrelevant to direct reuse on the benchmark, isolating the question of whether the architecture provides benefit through channels other than skill substitution. We return to this in the findings.

**5.4 Scoring**

Each benchmark task is scored on a 0-3 scale by score.py . An LLM-as-judge component (using the same Gemini model, with a different prompt) awards 0, 1, or 2 based on whether the agent's response is wrong/irrelevant, partially correct but vague, or correct/specific/grounded in real code. Three additional one-point bonuses are available — for code that successfully executes when extracted, for references to real files in the target repository, and for invocation of a verified skill from the library — but the final score is capped at 3 to avoid double-counting different forms of correctness.

The judge prompt and rubric are identical across conditions and tasks. Judge calls share the same model as the agent, so judge performance is bounded above by model capability; this is a known limitation of LLM-as-judge protocols \[@zheng\_judging\_2023\]. We discuss its implications in Section 9\.

**5.5 Aggregation**

For each tier, scores are averaged across tasks. For each condition, an overall mean is computed across the entire benchmark. The headline metric is the relative improvement of warm over cold: (warm\_avg \- cold\_avg) / cold\_avg × 100% .

**5.6 Quantitative Results**

Across the six-task benchmark:

| Tier | Cold avg | Warm avg | Difference |
| :---: | :---: | :---: | :---: |
| Tier 1 (Structural) | 1.50 | 2.50 | \+1.00 |
| Tier 2 (Pattern) | 2.00 | 2.50 | \+0.50 |
| Tier 3 (Novel) | 2.50 | 3.00 | \+0.50 |
| Overall | 2.00 | 2.67 | \+0.67 |

The relative improvement is \+33.3%.

Six skills were verified during the exploration phase. One of them (abstract\_collections\_abc\_imports ) was directly invoked during the warm benchmark and scored 3.0 out of 3.0 on its task. No skill compositions were observed during the warm run.

In the cold condition, the failure-classifier distribution was: missing-context (3), hallucination (2), tool-misuse (1); recovery succeeded on 50% of attempts. 

In the warm condition: missing context (2), hallucination (2), tool-misuse (1); recovery succeeded on 20% of attempts. 

The reduction in missing\_context failures from cold to warm is consistent with the system prompt being more informative in the warm condition (additional skills, episodes, and tool manifests are present in context) even when those skills are not directly invoked.

Sample size is small (n=6 benchmark tasks) and results derive from a single experimental run. We interpret these numbers as directional rather than statistically significant. The finding is reported with that caveat throughout.

6. **Responsible AI Considerations**  
     
   **6.1 Hallucination Mitigation**  
     
   The architecture minimizes hallucinations through three redundant layers:  
     
   Verification Pipeline: Functions must pass a four-stage gate, including LLM-authored unit tests and adversarial validation.  
     
   Episodic Memory: Injecting the three most similar past outcomes into the system prompt provides a "historical record" of successful strategies.  
     
   RAG Layer: Direct retrieval from authoritative ISO 20022 and SWIFT documentation reduces reliance on outdated pre-training data.  
     
   **6.2 Privacy and PII Handling**  
     
   The ComplianceShield serves as a conservative static analysis gate. It automatically rejects skills containing hard-coded PII patterns (e.g., SSNs, IBANs, or API keys). While effective for research, this regex-based approach currently lacks AST-level taint analysis to track variable provenance, a necessary requirement for production-grade deployment.  
     
   **6.3 Bias and Fairness**  
   We identify three primary sources of potential bias:  
     
   LLM-as-Judge: The evaluation scoring inherits the underlying model’s internal biases and may favor its own output style.  
     
   Classification Taxonomy: The five-category failure framework may misclassify edge cases not envisioned during design.  
     
   Embedding Bias: The all-MiniLM-L6-v2 model is trained on English-centric data, potentially impacting retrieval relevance for non-English identifiers or comments.  
     
   **6.4 Safety and Security**  
     
   To prevent the execution of malicious or runaway code, we implement:  
     
   Execution Gating: Mandatory unit testing and a "TEST PASSED" string requirement.  
     
   Orchestration Constraints: A 30-second subprocess timeout and restricted working directories.  
     
   Domain Safety: In financial contexts, the system blocks hard-coded routing numbers to prevent accidental money movement. For production, we recommend full containerization (e.g., E2B or Modal sandboxes).

7. **Findings and Discussion**

**7.1 The Multi-Channel Benefit Hypothesis**

The central empirical finding is that warm performance exceeded cold performance across all three task tiers, including the structural-retrieval tier (Tier 1, \+1.00), where the warm condition's accumulated skills should have been least directly applicable. This is the strongest single signal in the data and warrants careful interpretation.

A naive reading of the cold–warm hypothesis would predict the largest gains where direct skill reuse is possible (Tier 2 pattern replication and Tier 3 novel synthesis) and the smallest gains on Tier 1, since structural retrieval is purely a matter of reading and reporting facts from the target repository. The observed pattern is the inverse: Tier 1 showed the largest gain. Combined with the observation that only one of six accumulated skills was directly invoked during the warm benchmark, this points to the architecture's benefit being multi-channel rather than concentrated in skill substitution.

We identify three plausible channels and provide partial evidence for each: Channel 1 — Episodic memory. Every new task has the three most semantically similar past episodes injected into its system prompt. In the warm condition, those past episodes include twenty exploration episodes and any earlier benchmark episodes within the run. The model therefore enters each warm task with a richer working context, including records of what worked and what did not on related tasks. The reduction in missing\_context failures from three (cold) to two (warm) is consistent with this: when the model has examples of how prior tasks were grounded, it requests context-fetching tools more appropriately and hallucinates fewer file paths.

Channel 2 — Tool-call discipline. The skill manifest and tool manifest are both injected into the  
system prompt. In the warm condition the skill manifest is non-empty, which appears to nudge  
the model toward the action-as-Python-function pattern more consistently. We observed in logs  
that warm tasks were more likely to result in file-reading via the read\_file tool before  
generating output, even when no skill was eventually used. The accumulated context implicitly demonstrates the expected behavioural pattern.

Channel 3 — Direct skill reuse. This is the channel the architecture was explicitly designed to  
produce, and the one that, in this run, did the least work. The skill abstract\_collections\_abc\_imports was used on a Tier 3 task and scored 3.0/3.0, contributing one of the warm condition's score points. With a benchmark and exploration set drawn from the same domain, we would expect this channel to contribute substantially more. A more careful experiment would isolate these channels through ablation: a warm condition with episodic memory disabled, another with the skill manifest hidden but skills available for  
retrieval, and so on. We did not run these ablations and identify them as immediate future work.

**7.2 The Domain Mismatch**

The exploration tasks for the reported run were ISO 20022 and ACH payment utilities; the benchmark tasks were FastAPI internals. The two have essentially zero direct vocabulary overlap. This was a methodological flaw — the exploration set should have been calibrated to the benchmark domain — but it has produced an unexpectedly informative result. If the warm condition outperforms cold by 33% even when the accumulated procedural knowledge is from the wrong domain, the architecture's benefit cannot be exclusively (or even primarily) from direct skill reuse. It must be from the surrounding context-quality channels. This is, in a sense, a stronger result than a perfectly matched domain would have produced: it suggests the architecture has value even when the skill library is not perfectly aligned with the next task. A future experiment should run both conditions: a domain-matched warm run (payment exploration → payment benchmark), to isolate skill-reuse benefit, and a domain mismatched run, to isolate context-quality benefit.

**7.3 Verification-Pipeline Behaviour** 

Across the run, six skills passed all four verification gates and entered the library. The breakdown of failures by gate is informative.

Gate 1 ( ast.parse ) failures occurred most often during the period when gemini-2.5-flashlite output was being truncated mid-generation due to max\_output\_tokens being set too low (initially 2000). 

After increasing the limit to 8000 for the main agent loop and 6000 for skill synthesis and evolution paths, AST failures reduced substantially. This is a model-specific calibration issue rather than an architectural one, but it points to a real operational hazard: a token-output limit set too low produces silent skill loss that looks like model failure.

Gate 2 (subprocess unit test) failures occurred when the model wrote tests using assert  
statements without an explicit print("TEST PASSED") . The system's strict requirement on the success marker rejected several skills that, on inspection, were correct. We chose to treat this as desired behaviour: a test that does not announce its success is a test that cannot be reliably distinguished from a test that did not run, and false negatives at this gate are preferable to false positives.

Gate 3 (ComplianceShield) issued one BLOCK during the run on a payment-domain skill that  
contained a literal nine-digit number in its test fixture. It issued WARN flags on three of the verified payment skills ( parse\_pain001\_fields , build\_pacs008, build\_pacs002\_acceptance ) because they reference IBAN and amount fields without including audit-logging calls. These skills persisted with the warning recorded in their metadata, which is the intended graceful-degradation behaviour for the shield's warn class. Gate 4 (adversarial validation) caught four skills that had passed Gates 1-3, with the validating model identifying edge-case failures the original model's test had not covered. These appear in the log as skill\_validation\_failed . The model used for validation was the same gemini-2.5-flash-lite , which is a conflict-of-interest concern: a stronger validating model would likely catch more failures.

**7.4 Recovery Behaviour** 

The failure classifier ran on every low-graded task and triggered a single recovery attempt. Recovery rates were 50% in the cold condition and 20% in the warm condition. The reduction is not, as it might first appear, an indictment of the warm condition; rather, it reflects that the warm condition had fewer failures overall (because more tasks succeeded on first attempt) and that the failures that did occur were of a harder variety. The five-type taxonomy is also coarse: a single category like missing\_context covers a wide range of underlying causes from "did not call read\_file " to "called read\_file on the wrong path", and the recovery prompt cannot distinguish them.

8.  **Conclusion and Future Work**   
   

This paper presents a self-constructing agent architecture in which an LLM agent accumulates a library of verified, executable Python skills as a by-product of normal task execution, governed by a four-gate verification pipeline and a lifecycle layer that composes, evolves, and decays skills.

The system is motivated by the November 2025 SWIFT ISO 20022 migration and the broader  
observation that financial-protocol integration knowledge does not currently compound across  
institutions or engagements.

In a controlled cold–warm experiment against the FastAPI codebase, the warm condition  
outperformed the cold condition by 33.3% on average, with gains across all three task tiers. The  
result is directional rather than statistically significant given the n=6 benchmark size, but the  
pattern of gains is informative: only one accumulated skill was directly invoked, suggesting that  
the architecture's benefit is multi-channel — partly from accumulated context quality, episodic  
memory, and tool-call discipline, and partly from direct skill reuse.

**8.1 Limitations**

We identify five limitations to the current implementation and study.

Sample size: The reported result is from six benchmark tasks in a single experimental run. Replication on a fifteen-task benchmark with multiple runs per condition is necessary to convert directional evidence into a quantitative claim.

Single model: All experimental runs used gemini-2.5-flash-lite . The verification pipeline,  
the failure classifier, and the LLM-as-judge are all instantiations of the same model, which  
introduces correlation in their failure modes. A multi-model evaluation (e.g. a stronger judge, a  
different validation agent) would tighten the result.

Domain mismatch: Exploration was payment-domain, benchmark was FastAPI. This  
unintentionally isolated the context-quality benefit but also means we have not directly tested the hypothesis under matched conditions. A matched-domain run is the most immediate next experiment.

No ablations: The three benefit channels identified in Section 7.1 are not separately tested. An  
ablation series — episodic memory disabled, skill retrieval disabled, RAG disabled — would attribute the \+33.3% gain to specific architectural components.

Static, regex-based ComplianceShield: The shield catches obvious patterns but cannot reason  
about variable provenance, runtime data flow, or whether logged data actually constitutes an  
audit trail. It is appropriate for a research artefact and inappropriate for a production deployment without significant hardening.

**8.2 Future Work**

Several directions follow naturally. Domain-matched evaluation with a payment benchmark. The clearest immediate experiment is to replace the FastAPI benchmark with the ISO 20022 / ACH benchmark already partially built in payment\_benchmark\_tasks.txt . With matched exploration and benchmark, direct skill reuse should rise and the benefit channels can be more cleanly attributed.

Ablation of the three benefit channels. Run the warm condition with each context-injection  
layer (episodic memory, skill index, RAG, tool manifest) independently disabled, observing pertier deltas to attribute the \+33.3% gain.

Skill federation across deployments. The current skill library is per-installation. The largest  
practical benefit of the architecture in a multi-institution deployment would come from sharing  
verified, anonymised skills across deployments — an institution joining the network gets the  
accumulated payment-protocol knowledge of every prior institution. This raises non-trivial  
questions about trust, provenance, and update propagation that are themselves a research  
direction.

Stronger ComplianceShield. Replace the regex shield with AST-level taint tracking and  
integration with a real PII detector, with a quarantine-rather-than-reject workflow for borderline  
skills.

Stable-across-evolution skill identifiers. When a skill evolves from v1 to v2, callers of the v1  
function may break. A version-resolution layer that lets composed skills bind to a stable  
identifier while transparently routing to the latest verified version would let the library evolve  
without breaking dependent skills.

The broader thesis the architecture argues for is that procedural knowledge in code-acting  
agents should be persistent, verified, and organised by lifecycle rather than ephemeral and  
conversational. The cold–warm result reported here is one piece of evidence in support of that  
thesis. The more substantial test will come from longer-running deployments against real  
codebases, where the library's lifecycle mechanics — composition, evolution, decay — have time  
to operate at scale.
