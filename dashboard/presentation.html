The Problem

Every agent starts from zero.

Current LLM-based agents repeat the same work indefinitely — no compounding, no learning, no institutional memory.

❌ Problem 1

Cold Start Every Time

Each invocation has no memory of prior solutions. Solved a FastAPI routing problem yesterday? Start from scratch today.

❌ Problem 2

No Domain Compounding

Domain knowledge — ISO 20022 XML namespaces, ACH routing checksums — must be re-derived on every call. Nothing sticks.

❌ Problem 3

Unverified Outputs

LLM responses are never tested against a real environment. Hallucinated APIs get silently accepted as correct.

Hypothesis
An agent that accumulates a verified skill library performs measurably better than one without.

The structural problem

A consultant spends three years writing ISO 20022 message parsers. They get fast, accurate, aware of edge cases. Then they leave. The next institution hires a new consultant. The same problems get re-solved from scratch.

Knowledge does not compound across institutions, projects, or people.

Why an Agent
An agent that accumulates verified payment-protocol skills is, in principle, a partial answer.
The skills it builds for one institution apply to the next. The knowledge does not walk out the door.

Research Questions

Three questions, one architecture to answer them.

RQ1 · Performance

Does an accumulated skill library make the agent measurably better?

Tested via cold-vs-warm benchmark on the FastAPI codebase across three task tiers.

RQ2 · Mechanism

Where does the benefit come from?

Direct skill reuse, accumulated context quality, episodic memory — or some combination?

RQ3 · Verification

Is the verification pipeline strict enough?

Trustworthy enough for regulated domains, without suppressing skill formation entirely.

Our Answer · in one sentence
A self-constructing agent that writes, verifies, and accumulates a permanent library of executable Python skills as a by-product of normal task execution — and reuses them through composition, evolution, and decay.
verified persistence lifecycle management domain-applicable to payments

Architecture

Six layers. One self-improving loop.

1
Task Intake + RAG Context
run.py / rag.py — Task arrives; 7 seeded financial-spec docs retrieved. Embedding fallback via sentence-transformers.
2
Skill Selection (2-Stage)
select_skills() — Keyword pre-filter → embedding cosine rank → LLM ranker. Dynamic top-k based on library size.
3
LLM Response + Code Execution
run_task() — Gemini 2.5 Flash Lite generates response (max_tokens=8000). EXECUTE blocks run in subprocess.
4
Skill Persistence Pipeline
materialize_py_skill() — AST parse → subprocess verify → ComplianceShield → ValidationAgent → save.
5
Failure Classification + Recovery
classify_failure() — 5 failure types. Automatic recovery attempt with augmented prompt.

Skill Anatomy

Skill Evolution

Auto-healing skills

If fail_count ≥ 2 and fail > success, the agent calls evolve_skill() — prompts the LLM to fix it, archives v1, persists v2.

Skill Decay

Automatic forgetting

Skills unused for 14+ days with relevance < 0.1 are auto-archived to skills_py/archive/ — the library stays lean.

The 5-gate verification pipeline — every skill passes all gates or is rejected

Experiment Design

Cold vs. Warm. A controlled benchmark.

1
Reset State
Wipe skills_py/ and episodic memory. Back up existing state to eval/backups/.
2
COLD Benchmark
Run all benchmark tasks with empty skill library. Score with LLM judge (0–2) + bonuses.
3
Exploration Phase
Run 20 exploration tasks to build the skill library. Skills verified and persisted during this phase.
4
WARM Benchmark
Re-run the same benchmark with the populated skill library. Compare scores.

Safety Layer

Two guardians before any skill is persisted.

ComplianceShield · static analysis

Runs after code verification, before disk write. Pure static analysis — no LLM calls, no execution.

PASSClean code, no sensitive literals detected
WARNTouches PII/financial data without audit hook — persists with flag
BLOCKHardcoded routing number, IBAN, API key, or SSN — rejected entirely
# BLOCK pattern examples re.compile(r'routing_number\s*=\s*["\'][0-9]{9}') re.compile(r'\bIBAN\s*=\s*["\'][A-Z]{2}\d+') re.compile(r'api_key\s*=\s*["\'].+["\']') re.compile(r'\b\d{3}-\d{2}-\d{4}\b') # SSN
ValidationAgent · adversarial LLM testing

Uses the same Gemini model to generate 6 adversarial edge-case tests and runs them. Catches bugs the original unit test missed.

• Empty / None inputs
• Boundary values (min/max)
• Very large values
• Negative numbers
• Unicode strings
• Type edge cases

Technical Stack

Built entirely in Python. Zero external agent frameworks.

LLM Layer

Gemini 2.5 Flash Lite

google-genai SDK. Exponential backoff retry for 503s. max_tokens=8000 for skill generation to prevent truncation.

Embeddings

sentence-transformers

all-MiniLM-L6-v2 for semantic skill selection and episode retrieval. Cosine similarity with keyword fallback.

Skill Storage

Python files + JSON index

skills_py/*.py with YAML-style header comments. index.json for fast metadata lookups.

RAG

7 Seeded Spec Docs

ISO 20022, NACHA ACH, SWIFT MX migration, hybrid address format, reconciliation patterns, PII compliance, ledger entry fields.

MCP Tools

4 Built-in + Agent-Built

read_file · search_code · run_python · list_skills. Agent registers new tools via ---MCP TOOL--- blocks.

Eval Harness

LLM-as-Judge + Bonuses

Gemini scores each response 0–2. Execution, file reference, and skill-usage bonuses push to max 3.0. Results saved to eval/results/scores.json.

Roadmap

Known Issues (Active)
Output Truncation
gemini-2.5-flash-lite cuts generated code at ~100 lines causing syntax errors in payment simulation. Fix: max_tokens 2000→8000 applied.
Domain Mismatch in Exploration
Simple payment utility exploration tasks rarely match FastAPI benchmark tasks. Only 1 of 6 skills used in warm benchmark.
ComplianceShield WARNs on ISO Skills
ISO 20022 skills touch IBAN/amount fields without audit hooks — persist with compliance_warn flag but not ideal.
